JavaScript Rendering and SEO

Understand how search engines handle JavaScript-rendered content and the SEO impact of CSR/SSR/SSG

How Search Engines Handle JavaScript

Google Use两阶段索引：先抓取初始 HTML，再将页面放入渲染队列执行 JavaScript。渲染队列可能有数天延迟，且不保证完整执行所有 JS。Bing 对 JS 渲染的支持更有限。

Common Causes（JS 渲染导致收录问题）

  • SPA（单页应用）初始 HTML 只有空 div，内容完全依赖 JS
  • 关键内容通过 AJAX/fetch 异步加载
  • Using client-side routing, server returns same HTML for all routes shell
  • JS 执行依赖用户交互（如滚动加载）
  • 第三方 JS 错误导致页面渲染失败

Troubleshooting Steps

  1. Use curl 获取页面初始 HTML，Check是否包含核心内容
  2. 在 GSC URL CheckTools中对比"已抓取的页面"和"实时测试"
  3. 禁用浏览器 JavaScript 后访问页面，观察可见内容
  4. Check Google 缓存版本是否包含完整内容
  5. 查看 GSC 抓取统计中的渲染错误
  6. Use Chrome DevTools 的 Coverage 面板分析 JS 依赖
  7. Check是否有 JS 错误阻止页面正常渲染

Rendering Approach Comparison

方案SEO 友好度首屏速度适用场景
SSG（静态生成）最佳最快内容不频繁变化的页面
SSR（服务端渲染）优秀快动态内容、个性化页面
ISR（增量静态再生）优秀快内容定期更新的页面
CSR（客户端渲染）差慢仅限登录后的应用界面

Fix Methods

  • 将关键页面从 CSR 迁移到 SSR 或 SSG
  • Use Next.js、Nuxt.js、Astro 等支持 SSR/SSG 的框架
  • 确保 title、meta description、H1 和正文在初始 HTML 中
  • 对无法迁移的 CSR 页面Use预渲染服务
  • 避免将 SEO 关键内容放在需要用户交互才能加载的区域

Verification

Use HTML Readability Check Tool 粘贴 curl 获取的初始 HTML 进行Check。在 GSC URL Check中确认渲染后的页面包含完整内容。
